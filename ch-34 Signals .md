# Process Signals in Linux

## What are Process Signals?

A **signal** is a software interrupt sent to a process to notify it that an event has occurred. Signals are used by the Linux kernel and users to control the behavior of processes.

For example, signals can:

* Stop a process.
* Terminate a process.
* Pause or resume a process.
* Reload configuration files.
* Notify a process about specific events.

> **Simple Definition:**
>
> A signal is a message sent to a process requesting it to perform an action.

---

# How Signals Work

```text
User/Kernel
     |
     v
  Signal
     |
     v
 Running Process
     |
     +--> Terminate
     +--> Stop
     +--> Continue
     +--> Reload
     +--> Ignore
```

# Common Linux Signals

| Signal  | Number | Description                  |
| ------- | ------ | ---------------------------- |
| SIGHUP  | 1      | Reload configuration         |
| SIGINT  | 2      | Interrupt process            |
| SIGQUIT | 3      | Quit and create core dump    |
| SIGKILL | 9      | Forcefully terminate         |
| SIGTERM | 15     | Gracefully terminate         |
| SIGCONT | 18     | Continue paused process      |
| SIGSTOP | 19     | Pause process                |
| SIGTSTP | 20     | Terminal stop (Ctrl+Z)       |
| SIGCHLD | 17     | Child process status changed |
| SIGUSR1 | 10     | User-defined signal 1        |
| SIGUSR2 | 12     | User-defined signal 2        |

---

# 1. SIGHUP (1)

## Purpose

Reload configuration without stopping the process.

Syntax:

```bash
kill -1 PID
```

Example:

```bash
kill -1 1500
```

## Production Use

Reload a web server after changing its configuration.

Example:

```text
Update Config
      |
      v
SIGHUP
      |
      v
Reload Settings
```

---

# 2. SIGINT (2)

## Purpose

Interrupt a running process.

Usually generated by:

```text
Ctrl + C
```

Example:

```bash
ping google.com
```

Press:

```text
Ctrl + C
```

The process stops.

Equivalent:

```bash
kill -2 PID
```

---

# 3. SIGQUIT (3)

## Purpose

Terminate the process and create a core dump.

Syntax:

```bash
kill -3 PID
```

Used for debugging application crashes.

---

# 4. SIGTERM (15)

## Purpose

Gracefully terminate a process.

Default signal:

```bash
kill PID
```

Equivalent:

```bash
kill -15 PID
```

## Production Use

Preferred way to stop services because the process can:

* Save files.
* Close connections.
* Release resources.

---

# 5. SIGKILL (9)

## Purpose

Forcefully terminate a process.

Syntax:

```bash
kill -9 PID
```

The process cannot ignore this signal.

## Production Use

Use only when:

* Application hangs.
* Process does not respond to SIGTERM.

---

# 6. SIGSTOP (19)

## Purpose

Pause a process.

Syntax:

```bash
kill -19 PID
```

Example:

```bash
kill -19 1500
```

Process execution stops temporarily.

---

# 7. SIGCONT (18)

## Purpose

Resume a stopped process.

Syntax:

```bash
kill -18 PID
```

Example:

```bash
kill -18 1500
```

---

# 8. SIGTSTP (20)

Generated by:

```text
Ctrl + Z
```

Temporarily suspends a foreground process.

Example:

```bash
sleep 100
```

Press:

```text
Ctrl + Z
```

Resume:

```bash
bg
```

or

```bash
fg
```

# Production Workflow

Suppose an application is unresponsive.

## Step 1

Find PID:

```bash
ps -ef | grep java
```

Example:

```text
java 3456
```

---

## Step 2

Graceful termination:

```bash
kill 3456
```

---

## Step 3

Check process:

```bash
ps -p 3456
```

---

## Step 4

If still running:

```bash
kill -9 3456
```

---

## Step 5

Restart application:

```bash
sudo systemctl restart app
```

# Signal Priority

Production administrators usually follow this order:

```text
SIGTERM
   |
   v
Check Process
   |
Still Running?
   |
   v
SIGKILL
```

Avoid using SIGKILL first.

---

# Real Production Example

Suppose Nginx configuration changes.

Reload:

```bash
kill -1 PID
```

or

```bash
sudo systemctl reload nginx
```

No downtime occurs.

---

Suppose Java application freezes.

Graceful stop:

```bash
kill PID
```

If unsuccessful:

```bash
kill -9 PID
```

Restart:

```bash
sudo systemctl restart app
```

## Difference between SIGTERM and SIGKILL?

| SIGTERM                 | SIGKILL               |
| ----------------------- | --------------------- |
| Graceful shutdown       | Immediate termination |
| Process can clean up    | Cannot clean up       |

# Summary

Process signals are a fundamental part of Linux process management.

## Most Common Production Signals

| Signal  | Use                                 |
| ------- | ----------------------------------- |
| SIGTERM | Gracefully stop services            |
| SIGKILL | Forcefully terminate hung processes |
| SIGHUP  | Reload application configuration    |
| SIGINT  | Interrupt running tasks             |
| SIGSTOP | Pause a process                     |
| SIGCONT | Resume a paused process             |

## Production Best Practice

1. Identify the process.
2. Send **SIGTERM**.
3. Verify the process exits cleanly.
4. Use **SIGKILL** only if the process is unresponsive.
5. Restart the service if required.

Linux administrators, DevOps engineers, and SRE teams use process signals daily for application management and troubleshooting.
